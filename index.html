TiDB基础架构与拓展架构

基础架构

  • 无状态的SQL计算引擎(TiDB)
    • 解析应用层发送的链接
  • 分布式的KV存储引擎(TiKV)
    • 通过raft构建副本，实现高可用解决方案
  • 调度引擎(PD)
    • 元数据管理
    • TSO

      基于TiSpark的拓展架构

  • 引入一个jar包供Spark调用

基于TiFlash的拓展架构

TiDB技术特性

  • 在线可拓展性
    • 在线扩容
    • 在线缩容
    • 自动数据rebalance
  • 高可用
    • share nothing架构，存储引擎与计算引擎不存在单点瓶颈
    • 基于raft复制协议，具有多副本强一致性，不存在复制延迟
    • 故障自恢复
    • 两地三中心的多活架构
  • 支持分布式事务
    • TiDB控制ACID
    • 不需要在应用中声明分片键
    • 支持一个请求进行全局查询
  • 实时HTAP
    • 行列混存技术架构，实时查询，实时分析
    • 自动识别请求类型
    • 返回强一致结果
    • 不需要ETL的过程

TiDB4.0改进

TiUP

  • 在线两步安装TiDB
  • 支持离线安装
  • 省略磁盘IO检测

    大事务支持

  • 3.0
    • 100MB限制
    • 单行数据不超过6MB
    • 每个事务最多支持300000行
  • 4.0
    • 10GB限制
    • 单行数据6MB

      临时表

  • 超过内存限制时将数据转存到硬盘内，从而支持大SQL的响应

    Dashboard

  • 方便查看集群状态
  • 查看热点分布
  • 快速定位慢SQL
  • 性能调优指导

    弹性调度

  • 与k8s相结合
  • 对热点增加副本，对非热点减少副本

TiDB1_4笔记

发表于 2020-12-16
本文字数： 473 阅读时长 ≈ 1 分钟

TiDB架构

TiDB

SQL层

  • 无状态的SQL层，客户端可以链接到任意实例
  • 支持mySQL协议
  • Cost-based优化器
  • 二级索引
  • 在线DDL

TiKV/TiFlash

存储层

数据根据范围切分，同一个范围的数据存在多个副本，通过Raft共识协议同步，保证强一致以及高可用

  • TiKV
    • 行式存储，适合事务处理
  • TiFlash
    • 列式存储，适合分析处理
    • 副本固定为Raft Learner，使得对事务处理的影响最小化
  • 通过优化器选择，实现OLAP与OLTP隔离

PD

Placement Driver

  • 存储集群的元数据
  • 调度和负载均衡
  • 分布全局单调递增的事务时间戳

TiSpark

SQL层

  • 将SparkSQL直接运行在TiKV上的OLAP解决方案
  • 直接在同一平台进行事务和分析两种工作

生态工具

TiUP

  • 单机部署、集群部署、组件下载、版本控制及分发
  • 简单易用

    Lightning

  • 全量导入工具

    Dumpling

  • 全量导出工具

    Fast Backup & Restore

  • 分布式备份及恢复
  • 备份恢复性能可水平拓展

    TiCDC

  • 数据同步工具
  • 秒级上下游数据同步
  • 具有将数据还原到跟上游任一天都一纸的能力
  • 开发数据协议，与多种生态对接

TiDB1.3笔记

发表于 2020-12-16
本文字数： 553 阅读时长 ≈ 1 分钟

早期TiDB

可以认为是一个容量无限大的单机mySQL

  • 在计算和存储层面的无限拓展，可以分开拓展
  • 兼容mySQL语法和协议
  • 对应用透明的数据分片策略
  • 强一致性的分布式事务
  • 分为TiDB、PD、TiKV
    • TiDB：无状态的SQL引擎
    • TiKV：分布式KV存储引擎
    • PD：主管原数据的存储和TiKV中数据的调度

      TiDB十分适合中台场景

  • 由于协议兼容，可以从各种mySQL库进行数据同步
  • 不需要数据分片，对应用透明
  • 实时汇总数据
  • 相当于可以将备机和中台合二为一

    TiSpark

  • 优点：
    • 将单节点TiDB计算能力扩展为多节点的并行计算
    • 提供更快更稳定的分布式计算框架
    • 无缝接入大数据生态
  • 缺点：
    • 并发低
    • 消耗大量的计算资源
    • 维护难度较高

      TiDB2.0+

  • 优化器从Basic Optimizer到RBO+CBO到Cascades Optimizer(WIP)
  • 执行器从火山模型到批量化执行再到向量化执行，以及更好的并发控制
  • 引入分区表，Index Merge，对复杂计算能力有巨大提升

    TiFlash

  • 通过Raft Lerner向列存引擎同步数据
  • 通过标签实现物理隔离，AP、TP不再相互影响
  • 接受TiDB及TiSpark的读请求，通过Raft Learner同步写请求

    今日TiDB

  • TiDB = HTAP
  • 既有行存也有列存，数据自动进行行列转换
  • 运行TP业务时，可以方便进行报表查询及数据处理

TiDB1.2笔记

发表于 2020-12-15 更新于 2020-12-16
本文字数： 454 阅读时长 ≈ 1 分钟

什么是HTAP

就是混合OLTP与OLAP。

  • OLTP
    • 行存，实时更新
    • 高并发
    • 一致性
    • 每次只访问一小部分
    • 只存当前数据
  • OLAP
    • 列存，批量更新
    • 低并发
    • 每次访问大量数据
    • 处理历史数据

一般来说，会拆成两个系统，数据需要在两个系统间转运，而这会带来很大的延迟以及数据丢失。

为何需要 HTAP

TP与AP的界限变得模糊，比如综合性查询平台，需要支持报表查询和高并发短查询；或者对在线数据进行实时的分析。

  • HTAP 架构简单，运维成本低
  • 达到实时查询和实时分析
  • 提升业务敏捷性

比如在双十一活动时，就需要存储数据，同时实时地分析数据。

HTAP难点

  • AP可拓展性容易实现，TP可拓展性难以实现
  • 需要支持两种存储格式
  • 需要避免一种工作对另外一种工作的算力影响
  • 需要整合TP与AP的存储方式，使其高度耦合，从而得到实时的数据

    TiDB的HTAP特性

  • 可拓展
  • 经历过交易场景的检验
  • 强大的分析引擎
  • 天然适合实时数据应用
  • TiDB4.0新增特性
    • 可实时更新的列存储引擎
    • 行存和列存使用不同的机器，没有干扰
    • 一致性的复制
    • 列存有向量化的引擎，拥有良好的性能
    • 优化器可以自行决定使用行存或者列存

TiDB1.1笔记

发表于 2020-12-15
本文字数： 453 阅读时长 ≈ 1 分钟

90年代以前

对数据库要求较低

  • 使用SQL交互
  • 以单机形式存在，需要自行备份数据防止磁盘损坏，所有数据共享磁盘、cpu和网络。
  • 处理OLTP请求（小的事务和小的查询为主）
  • 典型的有PostgreSQL、MySQL、SQLServer

2000年后

互联网大爆发，数据量暴增，对数据库要求提高

  • 数据库中间件，如ShardingSphere、Vitess
  • 大数据平台出现（Hadoop）
    • 不支持SQL交互
    • 使用编程语言交互
  • 基于SQL的OLAP数据库（Greenplum、monetdb）
    • 联机分析处理系统，着重于处理海量数据的分析需求
    • 使用SQL进行交互
    • 分布式结构
    • 按列存储，处理分析查询更快
  • NoSQL
    • 使用key-value API交互
    • 放弃了关系、模型和事务
    • 高性能
    • 高可用和可扩展性
    • 分布式结构

      2010年后

  • NewSQL
    • 对OLTP负载提供了与NoSQL一样的可拓展性
    • 支持ACID事务
    • 使用SQL交互
    • 分布式
  • HTAP
    • 可以像NewSQL一样执行OLTP请求
    • 可以像数据仓库一样执行OLAP请求
  • TiDB
    • 高可用高拓展性
    • 支持OLTP和OLAP
    • 分布式
    • 使用SQL交互
